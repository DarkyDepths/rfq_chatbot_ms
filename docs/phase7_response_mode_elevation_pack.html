1.2 What this Pack adds

Phase 7B is not a rewrite. It is a targeted elevation of the existing response-mode machinery. Three new capabilities, three file extensions, zero new files created.

Phase 7B scope — three moves

Move 1: Make tool selection mode-driven. tool_controller.py currently selects tools by keyword independently of the response mode. After 7B, the response mode determines which tools to call and how much data to inject.

Move 2: Add evidence evaluation before rendering. A new method in rfq_response_controller.py checks whether tool results are sufficient for the chosen mode and downgrades honestly when evidence is lacking.

Move 3: Complete the trace logging. Every turn logs a structured JSON trace object covering intent → response mode → tool plan → evidence evaluation → grounding result.

Section 2

Frozen Decisions

Every decision in this section is final. No modification during implementation without explicit Pack amendment.

FD-7B-1 · Response mode enum stays as-is

The four RFQ response modes remain: FACT_FIELD, RFQ_SUMMARY, RFQ_DETAIL, RFQ_ADVISORY. The existing RfqResponseMode Literal type (L21 of rfq_response_controller.py) is not extended. The three non-RFQ modes (DOMAIN_KNOWLEDGE, OUT_OF_SCOPE, COURTESY) continue to be handled by intent routing in chat_controller.py — they never reach the response controller.

FD-7B-2 · Response mode selection stays in rfq_response_controller

The _select_response_mode() method (L333–348) remains the single authority for mode selection. It is not moved to a new controller. Reason: it already exists, works, and is deterministic. Creating a separate response_mode_controller.py would split logic unnecessarily. The method is extended, not relocated.

FD-7B-3 · Tool selection becomes mode-aware

Currently, tool_controller._plan_tool_use() selects tools by keyword matching, independently of the response mode. After 7B, the rfq_response_controller calls tools via the tool controller using a mode-driven tool plan — not keyword matching. The existing keyword logic in tool_controller remains as a fallback for turns that don't go through the response controller (unsupported, disambiguation).

FD-7B-4 · Evidence evaluation is a pre-render gate

A new method _evaluate_evidence() in rfq_response_controller.py runs between tool execution and _render_for_mode(). If the evidence is insufficient for the selected mode, the mode is downgraded (e.g., RFQ_ADVISORY → RFQ_DETAIL with an honest caveat). The downgrade is deterministic and logged.

FD-7B-5 · No new LLM calls

Phase 7B adds zero new Azure OpenAI calls. Response mode selection is deterministic. Evidence evaluation is deterministic. Tool planning is deterministic. The only LLM call per turn remains the existing generation call in _generate_and_persist_turn().

FD-7B-6 · The rfq_specific path bypasses LLM generation for grounded responses

This is the existing behavior and it stays. When _handle_rfq_specific delegates to rfq_response_controller.compose_response(), the assistant text is composed deterministically. The LLM is NOT called for rfq_specific turns that go through the response controller. This is a strength, not a gap.

FD-7B-7 · Trace schema is additive

New trace fields are added to existing log lines. No existing log fields are removed or renamed. The trace object is logged as a single structured JSON line at the end of _handle_rfq_specific().

Tool-Mode Matrix

Which tools each response mode requires, and what evidence check applies.

Response modeRequired toolsOptional toolsEvidence sufficient whenDowngrade target
FACT_FIELD get_rfq_profile get_rfq_stage (if stage-related field) Requested field exists and is non-null in unified view → RFQ_SUMMARY (if field not found, fallback to summary — this already happens at L396)
RFQ_SUMMARY get_rfq_profile get_rfq_snapshot At least one source ref present (profile or snapshot) → grounding gap (no source refs = "I don't have grounded RFQ data available right now")
RFQ_DETAIL get_rfq_profile + get_rfq_stage get_rfq_snapshot Profile source ref present + at least one additional source (stage or snapshot) → RFQ_SUMMARY (if only profile available, downgrade to summary depth)
RFQ_ADVISORY get_rfq_profile + get_rfq_snapshot get_rfq_stage Snapshot source ref present (snapshot is the primary advisory evidence source) → RFQ_DETAIL with caveat ("I can show current RFQ state, but grounded advisory signals require intelligence data that isn't fully available yet.")
Implementation note

This matrix is codified as a dict constant in rfq_response_controller.py. It is NOT a separate config file. Reason: the matrix is tightly coupled to the mode enum and the rendering logic — separating it would create drift risk between the matrix and the code that reads it.

Section 4

Evidence Evaluation Contract

The new pre-render gate that prevents fake intelligence and shallow restatement.

4.1 Data contract

# New dataclass in rfq_response_controller.py

@dataclass(frozen=True)
class EvidenceEvaluation:
    """Pre-render evidence sufficiency check."""
    original_mode: RfqResponseMode
    effective_mode: RfqResponseMode       # may differ if downgraded
    sufficient: bool
    downgrade_reason: str | None = None  # human-readable, logged
    caveat_text: str | None = None       # injected into response if downgraded

4.2 Evaluation rules (deterministic)

# Method: _evaluate_evidence(selection, unified_view) -> EvidenceEvaluation

# Rule 1: FACT_FIELD — check if the requested field is non-null
#   Already handled by existing _render_fact_field fallback at L396
#   No change needed — existing behavior is correct

# Rule 2: RFQ_SUMMARY — check if any source ref exists
#   Already handled by existing _render_summary at L511-512
#   No change needed — existing behavior is correct

# Rule 3: RFQ_DETAIL — check if profile + at least one more source
if selection.mode == "RFQ_DETAIL":
    has_profile = unified_view.profile_source_ref is not None
    has_stage = unified_view.stage_source_ref is not None
    has_snapshot = unified_view.snapshot_source_ref is not None
    if has_profile and not (has_stage or has_snapshot):
        → downgrade to RFQ_SUMMARY
        caveat = None  # summary is still useful, no caveat needed

# Rule 4: RFQ_ADVISORY — check if snapshot exists
if selection.mode == "RFQ_ADVISORY":
    has_snapshot = unified_view.snapshot_source_ref is not None
    if not has_snapshot:
        → downgrade to RFQ_DETAIL
        caveat = "Grounded advisory signals require intelligence snapshot data "
                 "that isn't fully available for this RFQ yet. "
                 "Here is the current operational state instead."

4.3 Integration point

The evaluation runs inside compose_response() between _select_response_mode() (L180) and _render_for_mode() (L181). The existing flow becomes:

# Current flow (rfq_response_controller.compose_response):
selection = self._select_response_mode(normalized_content)
assistant_text, source_refs = self._render_for_mode(selection, ...)

# New flow:
selection = self._select_response_mode(normalized_content)
evidence = self._evaluate_evidence(selection, unified_view)
effective_selection = ResponseModeSelection(mode=evidence.effective_mode,
                                             advisory_focus=selection.advisory_focus)
assistant_text, source_refs = self._render_for_mode(effective_selection, ...)
if evidence.caveat_text and assistant_text:
    assistant_text = f"{evidence.caveat_text}\n\n{assistant_text}"
Section 5

Mode-Driven Tool Selection

Making tool calls predictable by response mode, not query keywords.

5.1 The architectural change

Currently in _handle_rfq_specific() (chat_controller.py L365–373), tool selection happens before response mode selection. The flow is:

intent → tool_controller (keywords) → rfq_response_controller (mode + render)

After 7B, the response controller receives the raw preloaded context and drives tool selection itself:

intent → rfq_response_controller (mode → tools → evidence → render)

5.2 What changes in _handle_rfq_specific

The existing tool_controller.maybe_execute_retrieval() call at L366 is replaced by a new rfq_response_controller.compose_response_with_retrieval() method that:

# New method: compose_response_with_retrieval()
# 1. Select response mode from user query (existing _select_response_mode)
# 2. Look up required tools from the tool-mode matrix
# 3. For each required tool NOT already in preloaded_tool_call_records:
#    → call tool_controller.execute_single_tool(tool_name, rfq_id)
# 4. Build unified view from all tool results
# 5. Evaluate evidence sufficiency
# 6. Render for effective mode
# 7. Return RfqResponsePlan with all metadata

5.3 What changes in tool_controller

A new public method execute_single_tool(tool_name, rfq_id, preloaded_rfq_detail) is extracted from the existing _execute_tool() private method. This allows the response controller to call specific tools by name without going through keyword matching. The existing maybe_execute_retrieval() remains unchanged for non-rfq-specific paths (unsupported, disambiguation).

5.4 Preloaded context handling

The first-turn preload (chat_controller L806–877) already fetches profile + snapshot. The response controller checks preloaded records before calling tools again: if get_rfq_profile is already in the preloaded records, don't call it again. This is a deduplication check, not a new concept — the existing _ensure_primary_profile_record() at L194–235 already does this for profiles.

6.3 RfqResponsePlan extension

The existing RfqResponsePlan dataclass (L94–101) is extended with the new trace fields. These are metadata-only — they don't affect the response content:

@dataclass(frozen=True)
class RfqResponsePlan:
    response_mode: RfqResponseMode           # effective mode (after any downgrade)
    assistant_text: str
    grounded: bool
    source_refs: list[SourceRef]
    tool_call_records: list[ToolCallRecord]
    # Phase 7B trace metadata
    original_mode: RfqResponseMode | None = None      # before downgrade, None if no downgrade
    evidence_sufficient: bool = True
    downgrade_reason: str | None = None
    tools_planned: tuple[str, ...] = ()
    tools_executed: tuple[str, ...] = ()
    tools_from_preload: tuple[str, ...] = ()

Section 9

Implementation Sequence

Order matters. Each step has a review checkpoint before the next begins.

Step 1 → Step 2 → Step 3 → Step 4 → Step 5

Step 1 — Evidence evaluation in rfq_response_controller

Add EvidenceEvaluation dataclass, TOOL_MODE_MATRIX constant, and _evaluate_evidence() method. Wire into existing compose_response() between mode selection and rendering. Extend RfqResponsePlan with trace fields. All existing tests must still pass.

Review checkpoint: Unit test — RFQ_ADVISORY query with no snapshot → returns downgraded RFQ_DETAIL plan with caveat. Existing golden set cases still pass.

Step 2 — extract execute_single_tool in tool_controller

Make _execute_tool() public as execute_single_tool(). No behavioral change. All existing tests must still pass.

Review checkpoint: Existing tests pass. New method callable by response controller.

Step 3 — compose_response_with_retrieval

New method in rfq_response_controller that accepts tool_controller reference + preloaded records, determines required tools from matrix, deduplicates against preload, executes missing tools, builds unified view, evaluates evidence, and renders. Returns enriched RfqResponsePlan.

Review checkpoint: Integration test — full compose_response_with_retrieval call with mocked connectors, verifying tool plan + evidence evaluation + correct rendering.

Step 4 — Wire into chat_controller

Replace the two-step tool+response flow in _handle_rfq_specific() with the single compose_response_with_retrieval() call. Add phase7b.turn_trace log line. Pass tool_controller to response controller constructor or method parameter.

Review checkpoint: Full smoke test — Postman rfq_specific queries return correct responses. Structured logs include phase7b fields. All golden set cases pass.

Step 5 — Golden set cases + final verification

Add the 4 new golden set cases (23–26). Extend harness with mode/evidence assertions. Run full golden set. Verify all 26 cases pass.

Review checkpoint: All 26 golden set cases green. Structured logs verified. Defense demo script runs clean.

Defense framing

When this endpoint exists, the chatbot's RFQ_ADVISORY mode will consume these structured signals instead of deriving advisory content from the raw snapshot. The current advisory rendering in _derive_advisory_sections() is a correct intermediate step — it already computes concerns, missing elements, and attention items from available data. The signals endpoint formalizes and enriches this computation at the intelligence layer where it architecturally belongs.
