Section 01

The Core Question

Every architecture brief starts by stating what the thing is for. This one starts by stating what it is not.

The question is not "is the copilot smart?"
Who owns the conversation layer of the platform — and what concerns actually live there?
The naive answer: the copilot is a chat window on top of intelligence. Paste Azure OpenAI in front of the parsers, write a system prompt, let the LLM pick tools, call it done. That is roughly BACAB's current real-estate chatbot pattern, and it is exactly the wrong default for a high-stakes industrial RFQ platform.

The honest answer: the copilot is the conversational adapter that owns a set of concerns nobody else in the platform should own — session state, intent detection, RFQ context resolution, stage awareness, role awareness, provenance injection, guardrails, graceful degradation, and proactive behavior. It orchestrates calls to rfq_manager_ms for operational facts and rfq_intelligence_ms for derived insight, but it is neither of those. It is the surface through which humans interrogate and explore the platform.

Naming — product surface vs. technical service

Freeze

The product surface visible to users and stakeholders is called RFQ Copilot. The technical microservice that delivers it is rfq_chatbot_ms. Throughout this brief, product-facing prose uses RFQ Copilot (or simply the copilot), while engineering-facing prose — file paths, service identifiers, the BACAB layered structure — uses rfq_chatbot_ms. This is a deliberate separation: the product naming can evolve without requiring a technical migration, and the technical identity is stable for deployment, DNS, Docker Compose, and the rest of the platform.

One-sentence definition

Freeze

RFQ Copilot (technically rfq_chatbot_ms) is a stage-aware, role-aware, provenance-grounded conversational copilot that operates in two modes — embedded RFQ copilot and global portfolio copilot — orchestrates calls to rfq_manager_ms and rfq_intelligence_ms, owns the conversation-layer concerns neither of those services should own, and degrades gracefully when intelligence is partial or absent.

What the copilot is not

  • Not a reimplementation of BACAB's real-estate chatbot pattern. That pattern is LLM-as-orchestrator with free tool-calling. It works when wrong answers are cheap (a user scrolls past a mismatched apartment). In an Aramco pressure-vessel bid, a wrong cost estimate is real money. The architectural posture is different.
  • Not the owner of intelligence. Intelligence lives in rfq_intelligence_ms, a shared service (see D-07). The copilot is one consumer among several. It must not duplicate, mirror, or hide intelligence behind a conversational wall.
  • Not a state-mutating agent (in Phase 1). Phase 1 is read-only by design. All state changes — saving scenarios, updating fields, sending communications — are Phase 2 work, gated by explicit human-in-the-loop confirmation when they arrive.
  • Not a multi-agent system. There is one copilot, not many. Alternatives like "one agent per RFQ" or "specialized sub-agents coordinating via A2A" were considered and rejected for Phase 1 on coordination-cost and architectural-waste grounds (see §11 Decision Log). One brain, two modes, many scoped conversations.
  • Not an autonomous "AI agent" of the generalized Web3-era sense. It is a workflow-constrained copilot with explicit boundaries, explicit provenance, and explicit confidence. The word "agent" in the 2026 literature now means many things; the RFQ Copilot is specifically a workflow-constrained copilot in the lineage of CoALA (Sumers et al. 2024) and production state-machine agent patterns.

Section 04

The Two Modes — RFQ Copilot and Global Copilot

Before introducing the four pillars, we need to describe the shape of the product. The RFQ Copilot operates in two conversational modes, backed by one service and one brain. This section is the architectural contract that shapes every other decision in this brief.

The product shape
One copilot, two modes, many scoped conversations.
There is exactly one copilot service (rfq_chatbot_ms). There is exactly one conversation engine. What differs between modes is the entry context, the tool surface, the allowable question scope, and the default data the copilot operates on — not the underlying architecture. Alternatives like "one agent per RFQ," "one agent per dashboard," or "separate microservices per surface" were considered and rejected: they create coordination cost, context drift, and architectural waste with no benefit that cannot be achieved through clean scoping within a single service.

4.1 Mode A — RFQ Copilot (embedded, primary mode)

Mode A

RFQ Copilot

The copilot is opened from within a specific RFQ in the UI, already attached to that RFQ's context. Every conversation in this mode is RFQ-scoped: session memory is isolated per (user_id, rfq_id) pair, and no cross-pollination with other RFQs ever happens.

Entry and binding

Triggered from the RFQ detail page in rfq_ui_ms. The frontend opens a copilot panel and passes an explicit rfq_id in the session creation request. The copilot binds that rfq_id to the session at creation time; the binding is immutable for the lifetime of that session. If the user wants to talk about a different RFQ, they navigate to it and open a new conversation.

Default context

Because the rfq_id is known at session creation, the copilot pre-loads a lightweight RFQ snapshot on the first turn — fetched via get_rfq_snapshot(rfq_id) from rfq_intelligence_ms, which returns the consumer-facing read model already assembled by the intelligence service. The snapshot is the default context for every turn in the conversation; the copilot does not need to re-resolve which RFQ the user means.

Allowed questions

  • RFQ-specific factual queries — stage, workbook totals, BOQ items, parsed values, package identity.
  • RFQ-specific analytical queries — "what's missing from the BOQ?", "are the standards compliant with Aramco SAES-D-001?", "what's the margin trend on this cost breakdown?"
  • General engineering explanations — "what is ASME Section VIII?", "what does PWHT mean?" (subject to the knowledge boundary rule in §6 Pillar 2).
  • Stage-appropriate what-if exploration (Priority 2, stretch scope — see §12).

Tool surface

The full RFQ-scoped tool set: get_rfq_snapshot, get_intelligence_briefing, get_workbook_review, get_workbook_profile, get_rfq_profile, get_rfq_stage, search_mr_documents, get_capability_status. Portfolio-level tools (stats, analytics, cross-RFQ queries) are not exposed in this mode, because they would be out of the conversation's explicit scope.

4.2 Mode B — Global Copilot (portfolio, secondary mode)

Mode B

Global / Portfolio Copilot

The copilot is opened from a portfolio surface — RFQ queue, executive dashboard, overview page. It answers cross-RFQ and portfolio-level questions, and can pivot into a specific RFQ via disambiguation when the conversation narrows.

Entry and binding

Triggered from non-RFQ-scoped surfaces in rfq_ui_ms: the RFQ queue page, the executive dashboard, the overview. No rfq_id is bound at session creation time. The session starts in portfolio state. It may transition to RFQ-bound state later in the conversation if the user narrows to a specific RFQ — at which point the tool surface and allowed questions shift to match.

Default context

The default context is portfolio metadata read from rfq_manager_ms's stats and analytics endpoints (/rfqs/stats, /rfqs/analytics, and the RFQ list with filters). The copilot does not pre-load any single RFQ's snapshot, because the conversation may be about many RFQs at once.

Allowed questions

  • Portfolio questions — "which RFQs are blocked?", "which are near deadline?", "which client has the most active RFQs?", "which should I look at first?"
  • Cross-RFQ comparison — "how does RFQ IF-25144 compare to our other pressure-vessel jobs?" (limited by cold-start data reality — see §8 Pillar 4).
  • RFQ resolution by name — "show me RFQ IF-25144" or "what stage is the Aramco vessel RFQ in?" Handled via disambiguation (§9.5).
  • General engineering explanations — same knowledge boundary as RFQ mode.

Tool surface

Portfolio-focused tools are primary: list_rfqs, get_rfq_stats, get_rfq_analytics, resolve_rfq_by_name. When the conversation pivots into a specific RFQ, the RFQ-scoped tools become available as well, exactly as in Mode A.

The pivot — Global → RFQ-bound

State transition

When a global-mode conversation narrows to a specific RFQ (e.g., "tell me more about IF-25144"), the session transitions from portfolio to rfq_bound state. The rfq_id is set on the session, the RFQ snapshot is loaded, and subsequent turns behave as if the conversation had started in RFQ Copilot mode. The transition is one-way within a single conversation: once a conversation is bound to an RFQ, it cannot unbind back to portfolio scope — the user would start a new conversation to return to portfolio mode. This is a deliberate simplification to keep session state predictable and auditable.

4.3 What both modes share

Everything that matters for reliability, safety, and honesty is identical across the two modes. The four pillars (§5–§8) apply to both. The grounding guardrail, the confidence-aware rendering, the role-aware generation, the stage-aware behavior — all shared. The supporting stack (§9) is shared. The BACAB layered structure (§10) is a single codebase. There is no duplicated logic between modes.

4.5 Why not separate services?

Rejected — one service per surface

Approach: ship rfq_copilot_embedded_ms for the in-RFQ panel and rfq_copilot_portfolio_ms for the dashboard. Each with its own tool set, prompts, and deployment.

Why rejected: duplicates the entire conversation engine, the guardrail stack, the context builder, the memory layer, the evaluation harness. Every bug fix and every new pillar improvement has to be made twice. Coordination cost is paid forever in exchange for no architectural benefit that cannot be achieved through mode-based scoping within one service.

✓ Chosen — one service, two modes

Approach: rfq_chatbot_ms exposes both surfaces. A session_mode field on the session record indicates rfq_bound, portfolio, or pending_pivot. Routes are distinguished by entry endpoint but flow through the same controllers, context builder, and tool layer.

Gain: one codebase, one eval suite, one deployment, one telemetry surface. The mode distinction is a runtime parameter, not a deployment boundary. This is the same philosophy as the manager's single service handling both the UI read path and the event-driven write path.

4.6 What this does not mean

A few important non-implications worth stating clearly, because they are easy to misread:

  • This is not multi-agent. There are no sub-agents, no orchestrator, no A2A protocol. There is one Azure OpenAI call path per turn. The "two modes" are a property of sessions, not a property of agent topologies.
  • This is not HITL on RFQ resolution. When the user says "show me the Aramco vessel RFQ," the copilot resolves via search-and-disambiguate (§9 new subsection). HITL is only for state-changing actions, not for understanding what the user meant.
  • This is not stage-agnostic. Once a conversation is RFQ-bound (either from the start in Mode A, or after a pivot in Mode B), Pillar 1 (§5) applies fully: the current RFQ's manager-native stage drives tool availability and context framing.
  • This is not two memory stores. Episodic memory is one table, scoped by (conversation_id, user_id, rfq_id?). Portfolio sessions have rfq_id=null; RFQ-bound sessions have a value. One schema, one datasource, one retrieval path.
Defense paragraph

The RFQ Copilot operates in two modes — an embedded RFQ copilot bound to a specific RFQ at session creation, and a global portfolio copilot that answers cross-RFQ questions and resolves to specific RFQs via disambiguation when the conversation narrows. Both modes are served by a single microservice (rfq_chatbot_ms) with a single conversation engine, a single eval suite, and a single BACAB-layered codebase. What differs between modes is entry context, default tool surface, and allowed intent categories — not the underlying architecture. This is a deliberate simplification over alternatives like one-agent-per-RFQ or separate-service-per-surface, both of which would multiply coordination cost without architectural benefit.

Section 05 · Pillar 1

Stage-Aware Workflow-Constrained Copilot Innovation 01

What the agent does is constrained by where the RFQ is in its lifecycle. This is the first of four pillars and the one that most clearly differentiates the copilot from the LLM-as-orchestrator pattern.

Pillar 01

Stage-Aware Conversation

The copilot knows which RFQ lifecycle stage is active and adapts its available tools, its retrieved context, and its allowable questions accordingly.

The Problem

In an LLM-as-orchestrator pattern, the agent has full access to every tool on every turn. The LLM decides, from raw prompt-and-history context, what to do. This works for low-stakes domains where wrong actions are cheap. It fails in two specific ways for high-stakes industrial RFQs:

  • Stage-inappropriate actions. A user asks "when did we submit this?" for an RFQ that is still in the intake stage. The LLM, seeing a get_submission_date tool in its tool list, calls it, gets nothing useful, and either hallucinates a date or produces an awkward non-answer. Neither is acceptable.
  • Unbounded tool surface. The LLM has no principled way to know that certain questions only make sense in certain stages. The prompt becomes a wall of conditional instructions the LLM must hold in working memory on every turn, and Chroma's 2025 study (the "context rot" paper) shows exactly what happens when prompts grow: accuracy falls.

The Pattern — Workflow-Constrained Agent

The copilot implements a pattern that is variously called state-machine agent, workflow-constrained agent, or finite-state dialog policy over LLMs. The pattern was popularized in production systems by LangGraph in 2024–2025 and has academic roots in Sumers et al.'s CoALA — Cognitive Architectures for Language Agents (TMLR 2024).

The core idea: you trade some flexibility for massive reliability and explainability gains by conditioning the agent's available actions on an explicit state machine. In our case, the state machine is the RFQ lifecycle itself.

How It Works Concretely

Stage truth comes from rfq_manager_ms, not from an abstract enum. The authoritative vocabulary lives in manager's schema: workflow ← reusable workflow template (e.g., "Aramco PV standard flow") stage_template ← stage definitions within a workflow rfq_stage ← live stage instances per RFQ, with progression state The copilot fetches the current rfq_stage for the bound RFQ via: GET /rfqs/{rfq_id}/stages (from rfq_manager_ms) This returns the actual stage records (not an enum), including the active stage's template metadata, captured field values, progression timestamps, and subtasks. Stage-specific copilot configuration is keyed on the stage_template identifier: stage_config = { "stage_template_id:intake_parsing": { allowed_intents: [...], relevant_tools: [...], stage_framing: "..." }, "stage_template_id:technical_review": { allowed_intents: [...], relevant_tools: [...], stage_framing: "..." }, ... } On every turn (for RFQ-bound conversations): 1. Fetch current live stage from manager via get_rfq_stage(rfq_id). 2. Read the active stage_template identifier from the response. 3. Look up stage-specific copilot config by template identifier. 4. ContextBuilder uses stage_framing text in the system prompt. 5. Tool layer exposes only relevant_tools for this stage. 6. Intent classifier validates the user's intent against allowed_intents. 7. If intent is out of stage → graceful redirect, not awkward failure. Key point: the state machine IS the manager workflow. The copilot does not invent stages. It reads them. Workflows are configurable in manager; the copilot adapts automatically as long as the stage_template identifiers are present in the copilot's stage_config map.

Example — the same question, two stages

Consider a user asking "what's missing from the BOQ?" on an RFQ whose workflow includes an intake_parsing stage and a later submission stage:

  • Active stage = intake_parsing: perfectly valid. The copilot calls get_workbook_review(rfq_id) on intelligence, receives the workbook review report comparing intake vs workbook completeness, and renders missing or low-confidence items with source references.
  • Active stage = submission: no longer a productive question — the bid is already out. The copilot redirects: "This RFQ has been submitted. The workbook review from the estimation phase is archived and available, but any revisions now would require a change request. Would you like to review the archived workbook review, or check the submission status?"

Notice what the copilot did not do: it did not call the tool and produce a stale answer. It did not refuse the question. It offered stage-appropriate alternatives. This is the gain of workflow constraint — and notice that the stage names (intake_parsing, submission) are illustrative placeholders. The actual stage identifiers come from whatever workflow template is configured in manager for that RFQ class. The copilot adapts as manager's workflow templates evolve, without code changes — only a configuration update to stage_config.

Optional: conversational buckets

For UX framing and reasoning purposes, the copilot may optionally maintain a mapping from the manager-native stage templates to broader conversational buckets — higher-level groupings like "pre-bid," "in-review," "post-submission." These are never authoritative; they are narrative shorthand used only in the copilot's system prompt for framing and in user-facing phrasing. Manager's rfq_stage remains the single source of truth. This mapping is a simple dict in config/stage_config.py and is completely optional — the copilot works fine without it.

Why Plain Python and Not LangGraph

Rejected for this service, this phase — LangGraph framework

Why rejected: not because LangGraph is an invalid pattern — it is genuinely the dominant production framework for state-machine agents in 2025–2026, and BACAB may legitimately adopt it for other services where the orchestration complexity demands it. Rejected here because this copilot's required workflow control is small and explicit: the state machine has one dimension (manager-native stage templates), one trigger (every turn), and one action (reconfigure tool surface + prompt framing). That complexity is naturally expressible in BACAB's controller-centric architecture without importing a graph framework.

Also considered and rejected: LangChain (superset framework with broader scope), LangSmith (tracing and eval tooling), Langflow (visual builder). All three are fine tools; none of them pays for itself against the solo-engineer time budget and the simplicity of the orchestration this service actually needs.

✓ Chosen — Plain Python state machine in controllers

Approach: a module controllers/stage_controller.py holds the stage-template → config dictionary and the resolution logic. Every RFQ-bound request flows through it. Pure functions, no framework magic, trivially testable, fast to explain in the defense.

Trade-off honestly owned: we cite LangGraph and CoALA (Sumers et al. 2024) in the thesis as the academic and industry patterns we are aligned with, and we explain why a plain-Python implementation is cheaper to test, explain, and maintain for this service in this phase. If orchestration complexity grows meaningfully in Phase 2 — parallel branches, retry graphs, compensating actions — LangGraph becomes a reasonable migration target. For Phase 1, it would be complexity debt without return.

What ships in Phase 1

  • controllers/stage_controller.py with the resolution logic that reads rfq_stage from manager and looks up the matching copilot config.
  • config/stage_config.py — a static Python dict keyed on manager's stage_template identifiers, with allowed_intents, relevant_tools, and stage_framing for each.
  • Coverage for the stage templates that appear in the demo RFQ's workflow — typically 3–5 templates, determined by what manager's seed data creates for the IF-25144 golden sample.
  • Integration with ContextBuilder — stage framing is injected into the stable prefix of the system prompt.
  • Integration with the tool layer — only stage-relevant tools are exposed to Azure OpenAI on each turn.
  • Integration with intent classification — out-of-stage intents trigger graceful redirects.
  • Golden-set cases that verify stage-appropriate behavior on at least 3 different stage templates (same RFQ, different points in its lifecycle).
  • Handling for Global Copilot (Mode B) sessions that are not yet RFQ-bound — no stage gating applies until the pivot.

Thesis-Defensible Framing

Defense paragraph

We implement a workflow-constrained agent pattern (cf. LangGraph production patterns; Sumers et al. 2024, Cognitive Architectures for Language Agents, TMLR) in which the agent's available actions and context are conditioned on the manager-native workflow state. The state machine is not an invention of the copilot — it is a direct read of rfq_manager_ms's workflow / stage_template / rfq_stage schema, via the existing REST API. Copilot behavior is configured per stage template, not per abstract enum label, so it adapts automatically as manager's workflows evolve. This provides reliability, auditability, and explainability gains over pure LLM-as-orchestrator patterns — at the cost of some flexibility — a trade-off we justify by the high-stakes nature of pressure vessel bid management at GHI, where the downside of wrong action substantially exceeds the upside of conversational flexibility.

Section 06 · Pillar 2

Grounded Generation with Provenance Innovation 02

What the agent says is constrained by what it can point to. Every factual claim carries a source reference back to a deterministic parser output or a typed API result.

Pillar 02

Grounded Generation with Provenance

Every numeric or factual claim in the copilot's output carries a source_ref pointing to its origin. Claims without a source are rejected by an output guardrail and retried.

The Problem — Hallucination in a High-Stakes Domain

LLMs hallucinate. This is not a bug to be fixed with a better model; it is a structural property of generative probabilistic systems. In low-stakes domains (recommending a restaurant, explaining a concept) a 2% hallucination rate is tolerable. In an Aramco pressure-vessel bid with six-figure sums moving on numeric facts, it is not.

The anti-hallucination strategy that actually works in production in 2025–2026 is grounded generation (also called attributed QA): force every factual claim to carry a pointer to its source, and reject claims that don't. The copilot becomes, in effect, a lookup with narration rather than "an LLM that might be right."

The Data Shape — Tool Result Envelopes

Every tool in the copilot returns a structured envelope, not a bare value. The envelope is a frozen contract across all tools:

# tools/common/envelope.py
class ToolResultEnvelope(BaseModel):
    value: Any                              # the actual payload
    source_ref: SourceRef                    # where it came from
    confidence: ConfidenceLevel              # deterministic | pattern_1_sample | absent
    validated_against: Optional[str]         # "1_sample" | "multi_sample" | None
    retrieved_at: datetime                   # when the tool ran

class SourceRef(BaseModel):
    system: str          # "intelligence" | "manager" | "package_parser" | "workbook_parser"
    artifact: str        # "IF-25144.xls" | "SA-AYPP-6-MR-022" | "canonical_project_profile"
    locator: str         # "Bid S sheet, cell G47" | "folder 02 / BOM.xlsx" | "stage=parsed"
    parsed_at: Optional[datetime]

The System Prompt — Forcing the Model to Cite

The LLM is instructed to render source references inline with every factual claim. The instruction is part of the stable prefix (cached by Azure OpenAI, see Pillar support-stack section) and is short, direct, unambiguous:

# Excerpt from the stable system prompt

For every numeric or factual claim in your response, you MUST include
a source reference immediately after the claim, in square brackets.

Examples:

  "The grand total is $1,247,350
   [IF-25144.xls, Bid S sheet, cell G47, parsed 2026-04-12]."

  "This project uses Saudi Aramco standards SAES-D-001 and SAMSS-057
   [SA-AYPP-6-MR-022, folder 06 Standards]."

If a tool result did not carry a source_ref, DO NOT render the value
as a claim. Ask for clarification or call a different tool.

The Output Guardrail — Enforcement, Not Suggestion

System-prompt instructions are not enough. LLMs, especially under token pressure, will drop instructions they deem decorative. The copilot enforces the citation rule with a programmatic post-processor that lives in the controller layer:

After every LLM response generation: 1. Extract all numeric values from the response (regex). 2. For each numeric value, check: is a source_ref pattern present within N characters of the value? 3. If yes → pass. 4. If no → reject the response, log the failure, re-prompt with a stricter instruction that explicitly calls out the missing source. 5. Retry budget: 2 attempts. After 2 failures, return a safe fallback: "I have the value but cannot confidently cite its source — please consult the underlying artifact directly." This is the grounding_guardrail and it lives in controllers/guardrail_controller.py.

Why This Is Stronger Than "Trust the LLM"

Rejected — prompt-only grounding

"Tell the LLM to cite sources in the system prompt. It's a smart model, it'll do it."

Why rejected: this is the pattern every 2023–2024 chatbot tutorial advocates, and it fails under real-world pressure. LLMs silently drop citation behavior when the prompt is long, when the context is dense, or when the question is phrased conversationally. There is no enforcement, no measurement, no recovery.

Chosen — enforced grounding with retry

Prompt instruction + programmatic guardrail + retry loop + safe fallback + logging. The LLM is asked to cite; if it forgets, the guardrail catches it; if the retry fails, the user gets a safe fallback instead of a hallucinated number.

Result: the failure mode is "the copilot occasionally refuses to answer" rather than "the copilot occasionally lies with confidence." For GHI, that trade is obvious.

The knowledge boundary — general vs. RFQ-specific

A question I expect the jury to probe: "if the user asks about ASME standards, does the copilot just use LLM knowledge?" The answer is yes, but only for general explanatory questions, and the boundary must be explicit and defended. Here is the rule:

A. General engineering knowledge

LLM may answer from model knowledge without tool calls.

Examples:

  • "What is ASME Section VIII?"
  • "What is PWHT?"
  • "What's the difference between TEMA and ASME?"
  • "What does a U-stamp signify?"

These are general explanatory questions. The model is trained on public engineering content and can answer them as well as a textbook. The copilot does not need to call tools — it would be wasteful and the answer would not be any better.

B. RFQ-specific decision-driving knowledge

Tools + sources required. No exceptions.

Examples:

  • "Which ASME standards apply to this RFQ?"
  • "Does this package require PWHT?"
  • "Is this vessel compliant with Aramco requirements?"
  • "What inspection standard is mentioned in this MR package?"

These are project-specific decision-driving questions. The answer must come from manager facts, intelligence artifacts, or document retrieval over the actual MR package — never from model knowledge alone. Every numeric or factual claim must carry a source_ref.

The boundary in one sentence

General concept → LLM may explain. Specific project decision → tools and sources required. The boundary is enforced by the intent classifier (§9.6): intents tagged as general_knowledge bypass the tool layer and go straight to a cite-minimal narration; intents tagged as anything else (rfq_factual, rfq_analytical, portfolio, rfq_resolution) must call at least one tool before generating a response, and the grounding guardrail checks accordingly.

Why this boundary is safe, not lazy

A reader might worry: "if the copilot can answer general questions from model knowledge, isn't that a hallucination risk?" The honest answer: yes, it is a controlled risk, and here is how we control it. First, general-knowledge answers are clearly labeled in the response ("This is a general explanation of PWHT — not specific to any RFQ in the system"). Second, any question that even touches an RFQ-specific decision is routed to the grounded path by the intent classifier, which is tuned to err on the side of tool use. Third, the golden-set evaluation suite includes cases that test both sides of the boundary — questions the copilot should answer without tool calls, and questions where skipping a tool call is a failure.

This is an important defense point because without this rule, you have two bad options: either force tool calls for every question (wasteful, often impossible — there is no tool that "returns the definition of PWHT") or let the LLM free-form everything (dangerous, fails the grounding pillar). The boundary is the third option, and it is the one serious production systems use.

Thesis-defensible framing

Defense paragraph

The copilot implements grounded generation with provenance as its anti-hallucination strategy, paired with an explicit knowledge-boundary rule: general engineering knowledge may be answered from model knowledge without tool calls (clearly labeled as general); RFQ-specific decision-driving knowledge must come from tool results with source_ref metadata. Every tool result is a structured envelope carrying provenance; the system prompt instructs the LLM to render citations inline with every factual claim; a programmatic post-processor verifies citation presence and retries on failure. This is the conversational analogue of the deterministic-first discipline already applied in the workbook and MR-package parsers in rfq_intelligence_ms (cf. attributed QA literature; Rashkin et al. 2023 on attribution; and the 2024–2025 production consensus on reducing hallucination in high-stakes domains through enforced grounding rather than prompt-only suggestion).

Section 07 · Pillar 3

Role-Aware, Access-Aware Answer Generation Innovation 03

The same factual question returns different answers for different users, driven by identity and role resolved through the current IAM seam. This is where access control stops being an API-layer concern and becomes a generated-content concern.

Pillar 03

Role-Aware / Access-Aware Answer Generation

Identity is resolved through the current IAM seam (shared with rfq_manager_ms). Role is determined. Role-specific framing, allowed data, and redaction rules are injected into the prompt and enforced in the output layer. Same question, different role, different answer — by design.

The Problem

In most LLM-powered enterprise chatbots, access control stops at the API layer: "can this user hit this endpoint?" — yes or no. That's appropriate for traditional applications where the API returns structured data and the UI renders it with role-appropriate widgets. It is not sufficient for a conversational system, for one simple reason:

The insight

A copilot does not return data. It returns generated natural-language narrations of data. The narration itself is a new artifact the access-control system has never seen before. "Can this user know this?" is not the same question as "can this user's API call succeed?".

Concretely: an Estimation Department Lead and an executive might both have API-level read access to a pressure-vessel RFQ's canonical profile. But when they ask "how's the margin on this?", they need different answers. The Estimation Department Lead owns cost analysis and needs the internal cost breakdown and the raw margin number. The executive needs the margin number plus the bid-strategy framing ("comfortable," "tight," "requires negotiation"). A junior engineer, even with API-level access to the same data, should not see the raw margin at all — they're exposed to sensitive financial information they have no operational need for.

RBAC + ABAC — The Practical Pattern

The copilot uses the industry-standard combination of Role-Based Access Control (RBAC) for broad strokes and Attribute-Based Access Control (ABAC) for edge cases. This is not a new access-control model — it's the honest pattern most enterprise systems actually implement once they are serious. The novelty is in what we access-control: generated content, not just API endpoints.

Confidence-Aware Graceful Degradation Innovation 04

Pillar 04

Confidence-Aware Graceful Degradation

Every tool result carries explicit confidence metadata. The copilot's rendering of that result is explicitly conditioned on the confidence level. The cold-start isn't hidden — it's surfaced, honestly, in every turn.

Why This Pillar Exists

This pillar directly addresses the biggest academic vulnerability of the project: intelligence is cold-started on one sample. The PFE jury will notice. The question is how we answer when they ask about it.

Bad answer (defensive)

"Intelligence is partial because we don't have enough data yet."

Sounds like a limitation we wish wasn't there. Apologetic. Treats the cold-start as a wound.

Good answer (confident)

"Intelligence is intentionally layered by confidence, and the copilot is designed to behave differently depending on what intelligence can and cannot say. The cold-start isn't hidden — it's surfaced in every interaction. This is an architectural principle, not a workaround."

Turns a weakness into a design feature. The jury respects this.

Graceful degradation is what makes the second answer true. Without this pillar, we are claiming we have graceful degradation and hoping it works. With this pillar, we have designed and built the mechanism that makes the claim honest.

The Three Confidence States

Every piece of information the copilot uses has one of three confidence classifications, determined by rfq_intelligence_ms and surfaced through the tool result envelope (see Pillar 2):

State A — Deterministic high confidence

The information came from a deterministic parser operating on a known-template input. Example: the grand total from IF-25144's Bid S sheet, cell G47, extracted by the frozen workbook parser with fixed rules, no LLM involved.

Chatbot behavior: state the value plainly. No hedging. "The grand total is $1,247,350 [IF-25144.xls, Bid S sheet, G47]."

State B — Pattern-based, 1-sample validated partial confidence

The information is derived from a rule, heuristic, or extraction path validated against only the one golden sample. Example: a detection rule that says "this looks like an Aramco pressure vessel project" based on keyword patterns observed in SA-AYPP-6-MR-022 folder 06.

Chatbot behavior: state the value with explicit provenance framing. "Based on document keywords in folder 06, this appears to be an Aramco pressure vessel project (pattern-based, validated against 1 sample)." The qualifier is mandatory, not decorative.

State C — Absent / cold-start no guessing

The information would come from a capability that doesn't exist yet because the data isn't there. Example: historical win-rate benchmarking, requires outcome data the system doesn't have.

Chatbot behavior: do not guess, do not deflect vaguely, do not hallucinate a plausible number. Explicitly surface the gap and name the condition under which the capability would exist: "Historical win-rate benchmarking isn't available yet — this is the first recorded Aramco pressure-vessel RFQ of this class. Once outcomes are recorded on ≥5 similar projects, I'll be able to compare win rates."

The killer move

The named condition in State C ("once outcomes are recorded on ≥5 similar projects") is what makes this pillar different from "the copilot apologizes for missing data." It's not apology — it's a contract with the future. It also gives BACAB a concrete business case for collecting structured outcome data: here's exactly what the copilot could do once the data exists.

How It Works Technically

1. Every tool returns a ToolResultEnvelope (see Pillar 2) with confidence and validated_against fields already populated. 2. rfq_intelligence_ms owns confidence classification. The copilot never decides confidence — it reads what intelligence returns. (This is a minor extension to intelligence's existing frozen contract: surface the confidence metadata intelligence already tracks internally.) 3. The system prompt instructs the LLM on how to render each state: - confidence == "deterministic" → state plainly with source_ref - confidence == "pattern_1_sample" → state with "validated against 1 sample" qualifier - confidence == "absent" → state explicitly + named future condition 4. A dedicated tool get_capability_status(topic) exists for questions the system can't confidently handle. It returns a structured explanation of why the capability is absent and what would be needed. This is the "speakable absence" tool. 5. The output guardrail (see Pillar 2) is extended: if a tool result said confidence: pattern_1_sample but the response omits the qualifier, the guardrail rejects and retries. Grounding and confidence are enforced by the same mechanism.

9.1 Context Engineering — The Discipline Above Everything

Context engineering is the systems discipline of deciding what ends up inside the LLM's context window on every single call — which system instructions, which conversation history, which retrieved documents, which tool definitions, which user state — and in what order and what form. The term was popularized by Andrej Karpathy (mid-2025) and Anthropic's engineering team (Effective Context Engineering for AI Agents, 2025); it replaces "prompt engineering" as the primary skill for building reliable LLM systems.

The empirical finding that motivates the discipline is sharp: more context makes LLMs dumber, not smarter. Chroma's 2025 study on 18 frontier models found every model degraded as input length grew, some dropping from 95% to 60% accuracy past a threshold. The phenomena are named: context rot (accuracy decay with length), lost-in-the-middle (middle-sequence tokens get ignored due to RoPE positional encoding), context pollution (too much redundant info distracts), and context collapse (iterative rewrites erode important details).

The four core moves — and how we operationalize each one

Move Purpose How the copilot does it
Write Persist outside the context window Session state lives in Postgres (see §9.2 Memory). Scratchpads for intermediate reasoning are stored and referenced by ID, not re-pasted every turn.
Select Retrieve only what's relevant now Stage-relevant tool definitions only; current RFQ profile only; last N turns only; document retrieval only for document-intent queries.
Compress Summarize or trim older content Conversation history past N turns is compressed into a "conversation summary so far" block. The last N turns remain verbatim.
Isolate Partition to prevent cross-contamination Scoped sessions. RFQ-bound sessions belong to one RFQ; portfolio sessions isolate to portfolio data. No cross-pollination between different conversations. Session isolation is enforced at the session_id → rfq_id (which can be null) binding level.

All four moves appear explicitly in the architecture. This is not name-dropping; every move has a corresponding implementation in the code, listed above. The discipline is operationalized, not decorative.

The ContextBuilder — where it all lives in code

A single class, controllers/context_builder.py, is responsible for assembling the prompt on every turn. It takes a TurnRequest and returns a PromptEnvelope with explicit token budgets for each piece:

# controllers/context_builder.py
class ContextBuilder:

    def build(self, req: TurnRequest) -> PromptEnvelope:
        # STABLE PREFIX — cacheable by Azure OpenAI
        system_prompt    = self._base_system_prompt()           # ~1200 tokens
        role_framing     = self._role_framing(req.role)          # ~200 tokens
        stage_framing    = self._stage_framing(req.stage)        # ~250 tokens
        tool_definitions = self._tools_for_stage(req.stage)     # ~800 tokens
        # All of the above is IDENTICAL across turns of the same conversation.
        # Azure OpenAI will hit the prompt cache on it.

        # VARIABLE SUFFIX
        if session.mode == 'rfq_bound':
            context_snapshot = self._rfq_profile(req.rfq_id)    # ~400 tokens
        else:
            context_snapshot = self._portfolio_context()        # portfolio mode data

        conv_history         = self._compressed_history(req)    # variable, bounded
        latest_user_turn     = req.user_message                 # the actual new input

        return PromptEnvelope(
            stable_prefix=self._assemble_prefix(
                system_prompt, role_framing, stage_framing, tool_definitions
            ),
            variable_suffix=self._assemble_suffix(
                context_snapshot, conv_history, latest_user_turn
            ),
            total_budget=6000,  # hard token budget, tracked
        )
Why the order matters

The stable prefix comes first. Always. This is the architectural design decision that makes Azure OpenAI prompt caching applicable — not a runtime optimization. If the ContextBuilder were to put the user turn at the top and the system prompt at the bottom, caching would never fire and every turn would pay full price. The ordering is a design constraint enforced in ContextBuilder.

9.2 Memory — Two Layers, Honestly

The 2025–2026 literature distinguishes four memory types (working, episodic, semantic, procedural) drawn from cognitive science (Tulving's taxonomy; Sumers et al. 2024 CoALA). Phase 1 implements two of the four. Here's why, and what each one is:

Type Status Description and Phase 1 decision
Working memory Phase 1 The current context window itself, managed by ContextBuilder. Small, fast, volatile. Holds the active conversation and the immediate task context.
Episodic memory Phase 1 A Postgres table chatbot_conversations storing {conversation_id, rfq_id, user_id, turn_number, role, content, timestamp, tool_calls, source_refs}. Retrieval is by exact match (conversation_id) and by time window. No embeddings, no vector search — just clean relational storage of what happened.
Semantic memory Future work Distilled, generalized knowledge ("user prefers metric units," "this role typically asks about margins first"). Consolidation from episodic to semantic is the hardest problem in the field (cf. MemAgents ICLR 2026 workshop; Sumers et al. 2024; the consolidation step is "particularly underserved" in current systems). Deferred until real usage data exists.
Procedural memory Future work Learned workflows, tool-use patterns, SOPs. Requires actual usage to distill. Mem0's v1.0.0 API (early 2026) added explicit support; we defer on the same grounds as semantic memory.
Honest scoping

A full four-layer memory architecture with automated episodic→semantic consolidation is a research project by itself. Doing it on cold-start data would be overclaiming. The system is designed so that episodic memory accumulates from day one, and once sufficient real sessions exist (estimated ≥50 distinct RFQ conversations), consolidation into semantic memory becomes meaningful and validatable. This is the same validated_against: 1_sample → multi_sample discipline applied to the memory layer.

No Mem0 or MemMachine dependency. Both are excellent frameworks in 2026, but plain Postgres is more defensible for Phase 1: every row has a known schema, every retrieval is explainable, no framework magic, no learning curve, no new infrastructure (filter C10). The academic citation path remains open — the thesis can still cite Mem0 / MemMachine / CoALA as the conceptual lineage.

The counter-intuitive finding when mapping the retrieval landscape to GHI: classic RAG is not the main retrieval path. The important information in an industrial RFQ is structured data, already parsed and assembled into artifacts by rfq_intelligence_ms. When a user asks "what's the grand total?", semantic search over a document would be the wrong tool. The right tool is a direct typed call to intelligence's consumer-facing read model — the snapshot.

Queries like "which RFQs are near deadline?" go to the portfolio path. Defense paragraph

Retrieval defaults to intelligence's structured read APIs — the snapshot, briefing, workbook profile, and workbook review. Portfolio queries default to manager's stats and analytics endpoints. Document RAG over MR package content is an explicit exception path, used only for genuinely unstructured queries like spec interpretation. This is a deliberate application of the deterministic-first principle already used in the intelligence service's parsers — chosen because the failure mode of semantic search on structured data (paraphrase-driven wrong matches) is unacceptable in a domain where numeric precision matters. Classic RAG is an exception path, not the primary retrieval strategy.

What we do not build: GraphRAG across multiple RFQs (insufficient data), historical similarity search (cold-start), HyDE / query decomposition (not needed for the narrow retrieval scope), hybrid lexical+semantic fusion (nice-to-have but out of time budget). All are named as future work.

9.4 Tool Interface — MCP-Shaped, Not MCP-Native

The tool interface is the most strategically consequential decision in this brief, because it determines how forward-compatible the copilot is with the broader 2026 agentic ecosystem. The architectural choices slide commits to "MCP-shaped function calling (Phase 2: native MCP)". Here is the full reasoning.

The landscape in April 2026

The Model Context Protocol (MCP), introduced by Anthropic in November 2024, became the universal standard for connecting AI systems to tools and data in 16 months. By March 2026 it had 97M+ monthly SDK downloads and 10,000+ public servers, with adoption from Anthropic, OpenAI, Google DeepMind, Microsoft, and AWS. In December 2025 Anthropic donated it to the Linux Foundation's new Agentic AI Foundation to keep it vendor-neutral. Azure OpenAI supports MCP via the Microsoft Copilot Studio integration and the OpenAI Agents SDK.

In plain terms: MCP is the industry's "USB-C for AI." Exposing a service as an MCP server means any MCP-compliant client — Claude Desktop, ChatGPT, Cursor, internal dashboards, future agents — can talk to it. Write once, connect to many.

The three options we considered

Option A — Full Native MCP

Build MCP servers wrapping manager and intelligence; copilot is an MCP client.

Pros: maximally current, vendor-neutral, forward-compatible (any MCP client can drive our services later).

Cons: adds real implementation work. We can't touch manager (C4), so we'd need an MCP adapter service in front of it. OAuth 2.1 enterprise auth is still maturing (Q2 2026 roadmap). Too much for the Phase 1 time budget.

Option B — Function Calling, No MCP

Build function-calling tools using Azure OpenAI's native support; no MCP framing.

Pros: zero impact on manager, fits BACAB layered pattern, easiest to ship.

Cons: the thesis framing loses the "MCP-aligned, forward-compatible" story. Loses a meaningful architectural differentiator for the defense.

✓ Option C — MCP-Shaped Function Calling

Build function-calling tools, but design the tool surface as if it were an MCP server. Same naming conventions, same structured schemas, same resource/tool/prompt distinction.

9.5 RFQ Disambiguation — Search, Don't Gate

A recurring pattern in Global Copilot mode (and occasionally in RFQ mode when the user references another RFQ): the user names an RFQ and the copilot has to figure out which one they mean. "Show me IF-25144," "what stage is the Aramco vessel RFQ in?", "how does this compare to the pressure vessel job from last quarter?" These queries require RFQ resolution before any RFQ-scoped tool can be called.

Critical distinction

Disambiguation is not HITL. HITL (human-in-the-loop) is for state-changing actions — saving, updating, sending. Disambiguation is for understanding what the user meant, which is a different concern entirely. If the copilot had to pause and ask "are you sure?" every time the user referenced an RFQ by name, the UX would be unusable. Instead, disambiguation is a runtime capability of the copilot itself, expressed through the intent layer and the resolve_rfq_by_name tool.

The three-tier resolution pattern

RFQ resolution follows a three-tier pattern based on the ambiguity of the user's reference. The tier is determined by the resolve_rfq_by_name tool, which performs a search against manager's RFQ list with appropriate heuristics:

Tier Condition Copilot behavior
Low ambiguity auto-pick Exactly one match, or one match with a confidence score above the high-confidence threshold Auto-pick the matched RFQ, state the assumption explicitly in the response, and proceed with the requested action. Example: "I found RFQ IF-25144 — the Aramco pressure vessel project created on Feb 12. Here's the current stage..."
Medium ambiguity suggest + confirm One strong candidate and one or more weak alternatives, or one match below the high-confidence threshold Suggest the most likely candidate, mention the alternatives briefly, let the user confirm. Example: "I think you mean RFQ IF-25144 (Aramco pressure vessel, Feb 2026). There's also IF-24987 (Aramco heat exchanger) that might match. Which one did you have in mind?"
High ambiguity offer choices Multiple comparable matches, or a broad descriptor like "the pressure vessel RFQ" with several plausible candidates Offer 2–3 human-readable choices with enough metadata for the user to pick. Do not auto-pick. Example: "Several RFQs match 'pressure vessel': (1) IF-25144 — Aramco Yanbu PV, Feb 2026; (2) IF-24987 — Aramco Jubail PV, Dec 2025; (3) IF-25031 — GHI internal PV redesign, Jan 2026. Which one?"

How it integrates with the intent layer

Intent classifier detects rfq_resolution intent, or any RFQ-specific intent when the conversation is not yet RFQ-bound (Mode B before pivot). Flow: 1. Extract the RFQ reference from the user turn (by name, by ID, by descriptor). 2. Call resolve_rfq_by_name(query) — returns ranked candidates with confidence. 3. Determine tier from the ranking: - 1 match at high confidence → low ambiguity - 1 match below threshold → medium ambiguity - 1 strong + weak alternatives → medium ambiguity - N comparable matches → high ambiguity 4. Render according to tier (auto-pick, suggest+confirm, or offer choices). 5. If auto-picked or confirmed, pivot the session to rfq_bound state (Mode B → Mode A transition described in §4.2).

Why this pattern is honest

Two honest things about this design worth surfacing explicitly in the defense:

  • The copilot can be wrong about what the user meant, and that's fine. In the low-ambiguity tier, the copilot auto-picks and states the assumption. If the user meant a different RFQ, they correct the copilot and the conversation continues. The key is that the assumption is always visible — no silent picks.
  • Cold-start affects this too. With only one real RFQ in the system (IF-25144), every reference trivially resolves to that one RFQ. The three-tier pattern is designed, tested with synthetic multi-RFQ fixtures, and ready to activate the moment real multi-RFQ data lands. This is flagged honestly in the Pillar 4 spirit — the capability exists; the data to stress-test it does not yet.
Defense paragraph

RFQ resolution is handled as a first-class runtime capability of the copilot, expressed through a three-tier search-and-disambiguate pattern (low / medium / high ambiguity) backed by the resolve_rfq_by_name tool. It is deliberately not a human-in-the-loop gate — HITL is reserved for state-changing actions, not for understanding what the user meant. Assumptions are always stated explicitly, so the user can correct a wrong pick. The pattern is validated against synthetic multi-RFQ fixtures in Phase 1 and activates naturally on real multi-RFQ data in Phase 2.

9.6 Guardrails — Three Layers

The guardrail stack has three layers, each enforcing a different concern. All three live in controllers/guardrail_controller.py and are applied on every turn.

Layer When What it does
Input guardrails Before the LLM sees the message (1) Intent classification: is this an in-scope RFQ question? (2) RFQ resolution: which RFQ does the user mean? (3) Basic PII / secret filter using Azure AI Content Safety. (4) Prompt-injection heuristics (detect "ignore previous instructions" patterns).
Output guardrails After LLM generation, before user sees (1) Grounding check: every numeric claim must have a source_ref nearby (Pillar 2). (2) Confidence check: qualifiers required for non-deterministic claims (Pillar 4). (3) Role check: no forbidden content for the current user's role (Pillar 3). Failures trigger retry-with-stricter-prompt (budget: 2 attempts).
Action guardrails Before any state-changing tool runs Phase 1 is read-only — no state-changing tools exist in Phase 1 tool list. The guardrail is present but currently always rejecting: any attempt to call a write-tool is blocked and logged. This gives us the HITL enforcement point ready for Phase 2 activation.

The safety posture — HOOTL / HOTL / HITL spectrum

Action type determines the human-in-the-loop tier:

  • HOOTL (Human Out Of The Loop) — read-only queries. Fully autonomous. Examples: "what's the grand total?", "what stage is this in?", "show me the BOQ." The vast majority of Phase 1 queries.
  • HOTL (Human On The Loop, stretch) — what-if scenario exploration. The agent runs freely against a sandboxed copy of the canonical profile, logs everything, never mutates real state. The human monitors via the audit log. Example: "what if labor costs go up 5%?"
  • HITL (Human In The Loop) — state-changing actions. Require explicit confirmation, logged, subject to role check. Phase 1 exposes zero HITL actions. The enforcement mechanism is built and tested, but nothing calls it. Phase 2 will add the first mutations.
Defense framing

The copilot's safety posture is a three-tier HITL/HOTL/HOOTL layering driven by action type, paired with grounded generation enforcing factual citations and confidence-aware rendering handling epistemic uncertainty. Phase 1 operates HOOTL (read) and optionally HOTL (what-if sandbox, stretch); HITL is implemented as a mechanism but gated closed — no state-changing actions are exposed to the LLM in Phase 1, by design.

9.7 Determinism — Structure Wherever Possible

This family is where most of the engineering effort lives, even though it's the least glamorous. The principle: LLMs are probabilistic, business systems aren't — force structure everywhere you can.

  • Structured outputs everywhere it's possible. Intent classification returns {intent: enum, confidence: float, rfq_id: str | null}. Tool argument generation is validated against Pydantic schemas. Final responses requiring post-processing (grounding, confidence) are generated with a structured envelope.
  • Intent classification as the first step of every turn. This is non-negotiable. The LLM does not free-form its way through every request. Classify first (one cheap, structured LLM call), route based on classification, then generate. This alone eliminates a large class of "LLM did something weird" failures.
  • Pydantic validation at every boundary. Inputs into the ContextBuilder, outputs from tools, responses from the LLM wherever a schema applies. Fail loud at the boundary, not silently later.
  • Deterministic-first, LLM-second. Same principle as the intelligence service's parsers: if something can be answered by a database lookup, answer it by looking up the database. Reserve free-form LLM generation for the narration layer — turning structured tool results into natural language.

10.3 A note on services/

Per-service decision

BACAB's layered pattern treats services/ as a layer whose use is a per-service decision. The current repositories illustrate the spectrum:

  • rfq_manager_ms uses routes → controllers → datasources directly. No services/ folder. Controllers are the orchestration layer.
  • rfq_intelligence_ms uses routes → controllers → services → datasources/connectors. The services/ layer is a required boundary, because intelligence has complex internal orchestration (parsers, artifact assembly, async pipelines) that benefits from a dedicated layer between HTTP and persistence.

For rfq_chatbot_ms in Phase 1, the required orchestration is simple enough to live in controllers — the pattern matches manager, not intelligence. Every operation has a clear single-controller home, and controllers are the orchestration layer. This is a per-service decision, not a BACAB-wide principle; if orchestration complexity grows in Phase 2 (for example, multi-turn scenario planning, async event-driven proactive pushes, or multi-source evidence combination), a services/ layer can be introduced later following intelligence's pattern without disrupting the rest of the architecture.

The one immediately visible deviation from manager's layout is the tools/ folder, which contains MCP-shaped function-calling tools. It sits alongside controllers/ and datasources/ as a dedicated concern because the tools are a distinct architectural surface — they are the LLM's view into the platform — and benefit from being organized together rather than scattered across controllers.

Section 13

Defense Demo Script

Five minutes on screen. A small number of questions chosen to exercise all four pillars visibly and verifiably. This is the part of the defense where the jury forms its lasting impression of the copilot — the entire architecture brief above exists to make these five minutes credible.

13.1 The narrative arc

The demo is not a random collection of queries. It tells a story in four beats, carefully sequenced so each beat introduces exactly one pillar visibly:

  1. Beat 1 — "The copilot knows what it knows." Deterministic answer to a deterministic question. Pillar 2 (grounding) and Pillar 4 state A in action.
  2. Beat 2 — "The copilot knows when it's inferring." Pattern-based answer with explicit qualifier. Pillar 4 state B in action.
  3. Beat 3 — "The copilot knows what it doesn't know." Honest absence with named future condition. Pillar 4 state C in action — the killer move.
  4. Beat 4 — "The copilot respects who's asking." Same RFQ, same question, two authenticated sessions as two different users, two different role-appropriate answers. Pillar 3 in action — the most memorable moment.

Beats 1–3 take roughly 60 seconds each, all in one conversation as the Estimation Department Lead. Beat 4 takes 90 seconds and requires opening a second browser session — a second authenticated user context — because role in this system is a property of authenticated identity, not a runtime parameter that can be flipped mid-conversation. Plus a 30-second framing preamble and a 30-second closing. Total: ~5 minutes.

Pillar 1 (stage-aware) is demonstrated implicitly throughout — the copilot's answers are shaped by the fact that IF-25144 is in a parsing / review stage, and the system prompt framing reflects that. The jury can verify it by checking the request/response telemetry if they wish.
