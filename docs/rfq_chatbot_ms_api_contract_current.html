2. API Contract

Base: /rfq-chatbot/v1 · Convention: /{ms_name}/{version}/{resource}/{id}/{action} · /health is unprefixed.

Health

1
GET /health ops / k8s probe
Returns { "status": "ok", "service": "rfq_chatbot_ms" }. Registered with include_in_schema=False in code; documented here for operational completeness.

Smoke

1
GET /rfq-chatbot/v1/smoke bootstrap verification
Returns { "status": "ok", "service": "rfq_chatbot_ms", "phase": "phase-6" }. Proves the v1 router stack is wired.

Sessions

3
POST /rfq-chatbot/v1/sessions chat entry
Creates either an RFQ-bound or portfolio (global) session. Body: { user_id, mode: "rfq"|"global", rfq_id?, role? }. Returns 201 with the persisted ChatbotSessionRead.
  • If mode == "rfq" then rfq_id is required (else 422).
  • If mode == "global" then rfq_id must be null/omitted (else 422).
  • role falls back to AUTH_BYPASS_ROLE when omitted.
  • Persisted mode is the internal value (rfq_bound / portfolio), not the entry-mode value.
GET /rfq-chatbot/v1/sessions/{session_id} resume chat
Fetch one session by id. Returns ChatbotSessionRead or 404.
POST /rfq-chatbot/v1/sessions/{session_id}/bind-rfq portfolio → rfq pivot
One-way bind a portfolio (or pending-pivot) session to a single RFQ. Body: { rfq_id }. Returns the updated session.
  • Allowed transitions: portfolio → rfq_bound, pending_pivot → rfq_bound.
  • If the session already has an rfq_id, returns 409.
  • If the session is already rfq_bound, returns 409.
  • There is no unbind or rebind endpoint.

Chat

2
POST /rfq-chatbot/v1/sessions/{session_id}/turn primary chat path
Submit one user turn. Body: { content }. Returns TurnResponse { conversation_id, turn_number, role, content, source_refs[] }.
  • Loads the session (404 if missing).
  • Gets-or-creates the single conversation for that session.
  • Plans retrieval from the user content using keyword routing (see Section 4 — retrieval tools & provenance).
  • If retrieval is planned, the session must be rfq_bound and session.rfq_id must parse as a UUID; human-readable RFQ codes such as IF-25144 raise 422.
  • Persists the user message, builds the prompt envelope, calls Azure OpenAI chat completions, persists the assistant message.
  • tool_calls are persisted but not surfaced on the response DTO. Only source_refs[] are returned.
Error mapping: 422 (ambiguous retrieval, missing or non-UUID rfq_id, validation), 429 (Azure OpenAI rate limit after retries), 503 (upstream failure), 504 (upstream timeout).
GET /rfq-chatbot/v1/conversations/{conversation_id} history readback
Returns the conversation and its messages ordered by turn_number. Each message exposes its persisted source_refs[]. Persisted tool_calls are intentionally not surfaced on this read DTO yet.

Uniform error shape

All error responses follow { "detail": string }, produced by the application's exception handlers in src/app.py:
  • RequestValidationError → 422 with "Validation failed: <loc>: <msg> | …".
  • NotFoundError → 404.
  • ConflictError → 409.
  • UnprocessableEntityError → 422.
  • RateLimitError → 429.
  • UpstreamServiceError → 503.
  • UpstreamTimeoutError → 504.
  • Unhandled Exception → 500 with "Internal server error".

3. Current Limitations & Intentionally Deferred Items

These constraints are enforced in code today. They are documented honestly so that consumers (UI, jury, future engineers) do not assume capabilities that are not yet present.
Retrieval requires a UUID-bound RFQ session. POST /sessions/{id}/turn only retrieves manager / intelligence facts when the session is rfq_bound and session.rfq_id parses as a UUID. Human-readable RFQ codes such as IF-25144 are explicitly rejected with 422.
RFQ binding is one-way and immutable. Once a session has an rfq_id, bind-rfq returns 409. There is no unbind or rebind endpoint; users start a new session to switch RFQs.
One conversation per session. The pipeline always reuses the earliest conversation for a session. Multiple parallel conversations per session, and conversation listing endpoints, are deferred.
Tool selection is keyword-based and single-tool-per-turn. Two matching tool families in one user turn raise 422 ("ambiguous"). Briefing, workbook review, analytics, portfolio search, grand-total, and final-price keywords are handled by capability-status messaging (absence guidance) — these tools are not implemented yet.
Other deferred items. Streaming chat; native Azure OpenAI function-calling (the connector accepts a tools parameter but the controller never sends it); JWT / IAM-backed authentication; correlation-ID middleware and a /metrics endpoint; surfacing persisted tool_calls on read DTOs; the pending_pivot session mode (defined in the enum but not produced by any current endpoint).

4. Retrieval — Tools & Provenance

Three read-only tools are wired, each scoped to a single RFQ id. Every tool result is wrapped in a ToolResultEnvelope carrying a SourceRef for provenance.
Tool Selected when user content matches… Backed by Confidence
get_rfq_profile deadline, owner, client, country, industry, priority, workflow, rfq code, rfq status, status of this rfq GET /rfq-manager/v1/rfqs/{rfq_id} deterministic
get_rfq_stage stage, current stage, blocked, blocker, progress, assigned team GET /rfq-manager/v1/rfqs/{rfq_id}/stages deterministic
get_rfq_snapshot snapshot, currently known, known about this rfq, current snapshot, snapshot summary, missing info, missing information GET /intelligence/v1/rfqs/{rfq_id}/snapshot pattern_1_sample (validated_against: 1_sample)
Unsupported briefing, workbook review, workbook profile, analytics, stats, list rfqs, portfolio, grand total, final price, estimation amount — Handled as capability-status absence guidance (no retrieval)

Provenance shape on assistant messages

When a turn is retrieval-backed, the assistant TurnResponse and the corresponding ConversationMessageRead include one or more SourceRef entries:
  • system — origin service, e.g. rfq_manager_ms or rfq_intelligence_ms.
  • artifact — logical artifact name, e.g. rfq, rfq_stages, rfq_intelligence_snapshot.
  • locator — the upstream path used to retrieve the artifact.
  • parsed_at — upstream artifact timestamp when known.
Turns answered without retrieval return an empty source_refs[].

5. Data Model

Three tables, intentionally normalized: user_id and rfq_id live on chatbot_sessions and are reached from messages through conversation_id → chatbot_conversations.session_id. Denormalized copies are deferred until a real query pattern justifies them.

Table summary

chatbot_sessions — id (UUID PK), user_id (str, indexed), rfq_id (str?, indexed), mode ∈ {rfq_bound, portfolio, pending_pivot} (indexed), role (str), created_at, updated_at.
chatbot_conversations — id (UUID PK), session_id (UUID FK → chatbot_sessions.id, ON DELETE CASCADE, indexed), created_at, updated_at.
chatbot_messages — id (UUID PK), conversation_id (UUID FK → chatbot_conversations.id, ON DELETE CASCADE, indexed), turn_number (int), role (str: "user" | "assistant"), content (text), tool_calls (JSON?), source_refs (JSON?), timestamp (indexed). Constraint: UNIQUE(conversation_id, turn_number).

Note on scope recovery. Per the Phase 1 migration docstring, message scope (which user, which RFQ) is recovered through joins: chatbot_messages.conversation_id → chatbot_conversations.session_id → chatbot_sessions.(user_id, rfq_id). Denormalized copies on chatbot_messages are deferred until a real query pattern justifies them.
